Project Overview

What the game is · Player count · Engine versions

What the game is

Campbound is a 2-player, 3D asymmetric co-op survival game. Its design philosophy, documented in the accompanying Market Research Report, addresses a core flaw in the survival genre: "Role Overlap," where both players perform identical tasks and cooperation becomes meaningless. Campbound's solution is forced specialization — two players occupy roles so distinct that neither can survive without the other:

RolePerspectiveCore ResponsibilityFailure State
The Keeper Camp management & strategy Maintain campfire heat, cook food, fortify the base Scout dies without warmth/food
The Gatherer Action & exploration Scavenge dry wood, forage ingredients, avoid wildlife Camp starves without resources

Target platform: PC (Steam), premium pricing at $19.99, aiming for a 7.5/10+ user score by prioritising the "Communication Loop" between the two players.

NetworkObject Inventory

Every object with a NetworkObject component · Scripts · Ownership

There is one networked object in the project: the Player prefab. It is registered in DefaultNetworkPrefabs.asset and assigned as NetworkManager.PlayerPrefab.

ObjectNetworkBehaviour ScriptsOwnership Model
Player.prefab PlayerMovement, ThirdPersonCamera, ClientNetworkTransform Client-owned (owner-authoritative)

Script Detail: ClientNetworkTransform

// Subclasses NGO's built-in NetworkTransform.
// Single override turns off server authority.
[DisallowMultipleComponent]
public class ClientNetworkTransform : NetworkTransform
{
    protected override bool OnIsServerAuthoritative() => false;
}

Returning false makes the owning client the writer of position/rotation. NGO's built-in delta-compression and snapshot system handles the broadcast to all other peers.

Script Detail: PlayerMovement

public class PlayerMovement : NetworkBehaviour
{
    public override void OnNetworkSpawn()
    {
        if (!IsOwner) { enabled = false; return; }  // Non-owners do nothing
        thirdPersonCamera = GetComponent<ThirdPersonCamera>();
    }

    private void Update()
    {
        // Read legacy Input, compute direction relative to camera facing
        Vector3 direction = (camForward * vertical + camRight * horizontal).normalized;
        transform.position += direction * moveSpeed * Time.deltaTime;   // Local mutation
        transform.rotation  = Quaternion.RotateTowards(...);             // Local mutation
    }
}

Script Detail: ThirdPersonCamera

public class ThirdPersonCamera : NetworkBehaviour
{
    public override void OnNetworkSpawn()
    {
        if (!IsOwner) { enabled = false; return; }  // No camera for non-owners
        Camera.main.gameObject.SetActive(false);       // Disable scene camera
        // Spawn a dedicated camera for this client
        GameObject camObj = new GameObject($"PlayerCamera_{OwnerClientId}");
        camObj.AddComponent<Camera>();
        Cursor.lockState = CursorLockMode.Locked;
    }

    // Exposes PlanarForward / PlanarRight for PlayerMovement to use
    public Vector3 PlanarForward { get; private set; }
    public Vector3 PlanarRight   { get; private set; }
}

Connection Flow Step by Step

Join · Spawn · Disconnect lifecycle

Joining the Session

Both players launch the game. SampleScene loads. The HUD canvas is visible with Host and Client buttons.

Player A presses "Host." NetworkUI.StartHost() calls NetworkManager.Singleton.StartHost(). UTP begins listening on 127.0.0.1:7777. Connection approval is off — all connections are accepted. The host's own Player prefab is spawned immediately. HideUI() is called, hiding the canvas.

Player B presses "Client." NetworkUI.StartClient() calls NetworkManager.Singleton.StartClient(). UTP connects to 127.0.0.1:7777.

NGO handshake completes. The host spawns a Player prefab for Player B on all connected machines. HideUI() is called on Player B's machine.

Both players are in-world. Each machine has two Player objects: one owned locally, one replicated. ClientNetworkTransform begins syncing positions at 30 Hz.

On Spawn (OnNetworkSpawn)

For each spawned Player object, NGO calls OnNetworkSpawn() on all NetworkBehaviour components:

MachinePlayerMovementThirdPersonCamera
Owning client Script stays enabled. Fetches reference to ThirdPersonCamera. Disables Main Camera. Spawns a new Camera GameObject. Locks cursor.
All other machines enabled = false. No input read, no movement. enabled = false. No camera created.

On Disconnect

  • If the host disconnects, all clients are kicked (no host migration is implemented).
  • When a Player object is despawned, ThirdPersonCamera.OnDestroy() fires. If IsOwner, it releases the cursor lock (CursorLockMode.None, Cursor.visible = true).
  • The HUD canvas is not re-shown after disconnect — this is a known limitation (see Section 7).

Input and Authority Model

Who handles input · Where movement is calculated · Cheating prevention

Who handles input

Input is read exclusively on the owning client inside PlayerMovement.Update() and ThirdPersonCamera.Update(). Both scripts exit early (enabled = false) on non-owner instances, so remote player objects never read local input. The project uses Unity's legacy UnityEngine.Input API — the new Input System package (com.unity.inputsystem) is not installed.

InputAxisConsumer
Move forward / back"Vertical"PlayerMovement.Update()
Move left / right"Horizontal"PlayerMovement.Update()
Camera yaw (left/right)"Mouse X"ThirdPersonCamera.Update()
Camera pitch (up/down)"Mouse Y"ThirdPersonCamera.Update()

Where movement is calculated

Movement is calculated entirely on the owning client. The data flow is:

// 1. Owner reads input axes (legacy UnityEngine.Input)
float horizontal = Input.GetAxis("Horizontal");
float vertical   = Input.GetAxis("Vertical");

// 2. Direction computed relative to the camera's yaw
Vector3 direction = (camForward * vertical + camRight * horizontal).normalized;

// 3. Transform mutated LOCALLY on the owner
transform.position += direction * moveSpeed * Time.deltaTime;

// 4. ClientNetworkTransform detects the delta and sends it to the server/other clients
//    (handled internally by NGO — no custom RPC needed)

The server never recalculates movement. It receives the position update from the owning client and replicates it to all other peers.

Cheating prevention

🚨

There is no cheating prevention in the current build. The owner-authoritative model means a malicious client can freely manipulate transform.position and every other machine will display that position faithfully. There is no server-side validation of movement speed, position delta, or physics.

This is acceptable for a course prototype where both players are trusted peers, but it would need to be replaced with server-authoritative movement or server-side position validation before any public release.

Known Limitations and Future Work

What doesn't work · What was cut · What comes next

What doesn't work yet

LimitationDetail
LAN-only connectivity Transport is hardcoded to 127.0.0.1:7777. Players on different machines or networks cannot connect without Unity Relay + Lobby.
No asymmetric roles Both players use the identical Player prefab with identical movement. The Keeper and Gatherer roles exist only in design documentation, not in code.
No HUD restore on disconnect After disconnect, the connection canvas is not re-enabled. The player is stuck on an empty scene.
No connection approval or matchmaking Any process that knows the IP can join. No lobby, room code, password, or session management exists.
No survival loop No campfire, resource, health, heat, or hunger system. The core gameplay loop from the design document is entirely absent.
No scene transitions Only one scene exists. No main menu, loading screen, lobby scene, or end-game screen.
No cheating prevention Owner-authoritative movement with no server validation (see Section 6).

What was cut for scope

  • Asymmetric perspectives — first-person view for The Gatherer and management-view for The Keeper were not implemented; both players share the same third-person camera.
  • Environmental hazards — wildlife, weather, and terrain danger described in the design are absent.
  • Inventory and resource systems — dry wood, food ingredients, and a resource economy were not built.
  • Fire mechanics and heat levels — The Keeper's core responsibility has no implementation.
  • AI Navigation — com.unity.ai.navigation 2.0.9 is listed as a dependency but no NavMesh agents, NPCs, or AI wildlife exist in the scene.

What would be added with more time

FeatureTechnical Approach
Internet connectivity Replace the hardcoded UTP loopback address with Unity Relay + Lobby. Players would create/join rooms with a shareable 6-digit code via Unity Gaming Services.
Asymmetric role prefabs Connection approval callback assigns a role (Keeper / Gatherer) on join and spawns a different prefab per role, each with different NetworkBehaviour components and camera rigs.
Server-authoritative movement Switch to default NetworkTransform (server-owned) or add a [ServerRpc] that validates position deltas against a max-speed threshold before applying them.
Game state with NetworkVariables A GameState NetworkBehaviour on the host would hold NetworkVariable<float> fields for campfire heat, food stock, and player health — automatically replicated to all clients.
RPCs for discrete events "Resource picked up", "fire extinguished", "player death" are one-time events well-suited to [ServerRpc] calls that update authoritative state and broadcast results via [ClientRpc].
Scene management Full flow: Main Menu → Lobby → Gameplay → Results, managed by NGO's built-in NetworkSceneManager for synchronized transitions.
New Input System Migrate from legacy Input.GetAxis to com.unity.inputsystem for proper per-device isolation, gamepad support, and cleaner ownership gating.
